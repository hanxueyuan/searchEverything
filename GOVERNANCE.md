## 角色与职责

### 👑 项目所有者 (Owner)

**职责**:
- 项目整体方向和战略规划
- 最终决策权（技术分歧、重大变更）
- 维护者团队的任命和授权
- 社区规范和争议的最终裁决

**权限**:
- 所有仓库的完全访问权限
- 合并任何 PR
- 发布任何版本
- 管理维护者团队

### 🔧 核心维护者 (Core Maintainers)

**职责**:
- 日常项目维护
- PR 审核和合并
- Issue 分类和分配
- 版本发布
- 社区管理

**权限**:
- 合并审核通过的 PR
- 关闭/重新打开 Issue
- 管理标签和里程碑
- 发布补丁和次要版本

**成为核心维护者的条件**:
- 持续贡献 3 个月以上
- 至少合并 10 个 PR
- 获得现有维护者团队提名和投票通过

### 👨‍💻 技术负责人 (Tech Lead)

**职责**:
- 技术架构设计和评审
- 代码质量标准制定
- 技术方案评估
- 指导开发者的技术工作

**权限**:
- 批准技术相关的 PR
- 决定技术方向
- 制定编码规范

### 🧪 QA 负责人 (QA Lead)

**职责**:
- 测试策略和计划
- 测试标准制定
- 发布质量审核
- QA 团队管理

**权限**:
- 批准测试相关的 PR
- 决定是否达到发布质量标准

### 👩‍💻 开发者 (Developers)

**职责**:
- 功能开发
- Bug 修复
- 代码审查
- 单元测试编写

**权限**:
- 提交 PR
- 参与技术讨论
- 审核他人代码（非最终批准）

### 🔍 测试工程师 (Test Engineers)

**职责**:
- 测试用例设计
- 自动化测试开发
- 手动测试执行
- 回归测试

**权限**:
- 验证 PR 的测试覆盖
- 标记测试相关问题
- 建议发布决策

### 🌟 社区贡献者 (Contributors)

**职责**:
- 提交 Issue 和 PR
- 参与讨论
- 帮助其他用户

**权限**:
- 提交 PR（需审核）

## Issue 处理流程

### Issue 分类

| 类型 | 标签 | 优先级 | 响应时间 | 处理 SLA |
|------|------|--------|----------|----------|
| 🐛 Bug - 严重 | `bug`, `critical` | P0 | 24 小时 | 3 天内修复 |
| 🐛 Bug - 普通 | `bug` | P1 | 48 小时 | 1 周内修复 |
| 🐛 Bug - 轻微 | `bug`, `minor` | P2 | 72 小时 | 2 周内修复 |
| 💡 功能请求 | `enhancement` | P2/P3 | 72 小时 | 评估后决定 |
| 📚 文档 | `documentation` | P2 | 72 小时 | 1 周内修复 |
| ❓ 问题 | `question` | P3 | 72 小时 | 回答后关闭 |
| 🧪 测试 | `testing` | P2 | 72 小时 | 1 周内处理 |

### Issue 标签系统

#### 类型标签
- `bug` - Bug 报告
- `enhancement` - 功能增强
- `documentation` - 文档相关
- `question` - 问题咨询
- `testing` - 测试相关

#### 优先级标签
- `P0` - 紧急（系统崩溃、数据丢失、安全漏洞）
- `P1` - 高优先级（核心功能不可用）
- `P2` - 中优先级（功能缺陷，有替代方案）
- `P3` - 低优先级（改进建议、锦上添花）

#### 状态标签
- `triage` - 待分类
- `confirmed` - 已确认
- `in progress` - 处理中
- `needs info` - 需要更多信息
- `blocked` - 被阻塞
- `ready for review` - 待审核
- `done` - 已完成

#### 特殊标签
- `good first issue` - 适合新手
- `help wanted` - 需要帮助
- `wontfix` - 不会修复
- `duplicate` - 重复 Issue
- `invalid` - 无效 Issue

### Issue 响应流程

```
1. 提交 Issue
   ↓
2. 自动添加 `triage` 标签
   ↓
3. 维护者分类（24-72 小时内）
   │
   ├─ 信息不足 → 添加 `needs info`，等待补充
   ├─ 重复 Issue → 添加 `duplicate`，关闭
   ├─ 无效 → 添加 `invalid`，关闭
   └─ 有效 → 添加适当标签，分配处理人
   ↓
4. 处理中 → 添加 `in progress`
   ↓
5. 修复完成 → 关联 PR，添加 `ready for review`
   ↓
6. PR 合并 → 自动关闭 Issue
```

### Issue 分配机制

1. **自动分配**:
   - 根据 Issue 类型自动分配给对应团队
   - Bug → Tech 团队
   - 文档 → 文档维护者
   - 测试 → QA 团队

2. **手动分配**:
   - 维护者根据工作负载手动分配
   - 贡献者可以主动认领（评论 "I'll work on this"）

3. **认领规则**:
   - 认领后 7 天内无进展，重新开放认领
   - 同一人最多同时认领 3 个 Issue
   - `good first issue` 优先分配给新贡献者

### Issue 关闭条件

Issue 在以下情况下可以关闭：

- [ ] 问题已修复且 PR 已合并
- [ ] 功能已实现且 PR 已合并
- [ ] 问题无法复现且超过 30 天无新信息
- [ ] 被标记为 `duplicate` 且原 Issue 仍存在
- [ ] 被标记为 `wontfix` 且有合理解释
- [ ] 用户自行关闭

## PR 审核流程

### PR 提交

1. 贡献者创建 PR，填写 PR 模板
2. 自动触发 CI 检查（测试、clippy、fmt）
3. 自动添加 `needs review` 标签

### 审核检查清单

审核人必须检查以下项目：

#### 代码质量
- [ ] 代码通过 `cargo clippy -- -D warnings`
- [ ] 代码通过 `cargo fmt --check`
- [ ] 无编译警告
- [ ] 遵循项目代码风格
- [ ] 无硬编码值（使用配置或常量）
- [ ] 错误处理适当

#### 测试覆盖
- [ ] 新增功能有单元测试
- [ ] 边界情况有测试
- [ ] 错误路径有测试
- [ ] 集成测试已更新（如需要）
- [ ] 所有测试通过
- [ ] 测试覆盖率未显著下降

#### 文档更新
- [ ] README 已更新（如有需要）
- [ ] 代码注释完整
- [ ] 公共 API 有文档注释
- [ ] CHANGELOG.md 已更新
- [ ] 示例代码已更新（如需要）

#### 向后兼容性
- [ ] 无破坏性变更，或
- [ ] 破坏性变更已明确标注并有迁移指南
- [ ] 配置变更有向后兼容处理

#### 安全性
- [ ] 无新的安全警告
- [ ] 输入验证适当
- [ ] 无敏感信息泄露
- [ ] 依赖项安全（无已知漏洞）

### 审核通过标准

**必须满足**:
- ✅ 所有 CI 检查通过
- ✅ 至少 1 名核心维护者审核批准
- ✅ 无未解决的评论
- ✅ 测试全部通过

**建议满足**:
- 📋 QA 团队验证通过（针对重大变更）
- 📋 文档完整
- 📋 性能无显著下降

### 审核角色要求

| PR 类型 | 最小审核人数 | 必须包含 |
|---------|-------------|----------|
| 文档修复 | 1 | 任何维护者 |
| Bug 修复 | 1 | Tech 团队成员 |
| 新功能 | 2 | Tech Lead + QA |
| 破坏性变更 | 2 | Owner/Core Maintainer |
| 核心模块变更 | 2 | Tech Lead + Core Maintainer |

### 合并策略

#### 默认策略：Squash and Merge

- 将多个提交压缩为单个提交
- 保持提交历史简洁
- 提交信息使用 PR 标题和描述

#### 其他策略

| 策略 | 使用场景 |
|------|----------|
| `squash` | 默认，多提交的功能开发 |
| `rebase` | 单提交的简单修复 |
| `merge` | 需要保留完整提交历史的大型功能 |

### 合并后流程

1. PR 合并后自动关闭
2. 触发 CI 构建和测试
3. 更新相关文档
4. 通知关联 Issue

### 冲突解决

#### 技术分歧处理

```
分歧发生
    ↓
双方讨论（24 小时内）
    ↓
无法达成一致？
    │
    ├─ 是 → 提交 Tech Lead 裁决
    │        ↓
    │     仍有分歧？
    │        │
    │        ├─ 是 → 提交 Owner 最终决定
    │        └─ 否 → 执行 Tech Lead 决定
    │
    └─ 否 → 达成共识，继续开发
```

#### 决策原则

1. **数据驱动**: 优先基于测试数据和性能指标决策
2. **用户优先**: 以用户体验和需求为最高优先级
3. **技术债务**: 避免引入新的技术债务
4. **简单优先**: 在满足需求的前提下，选择更简单的方案
5. **向后兼容**: 尽量避免破坏性变更

#### 最终决策机制

| 决策类型 | 决策者 | 咨询对象 |
|---------|--------|----------|
| 日常技术决策 | Tech Lead | 相关开发者 |
| 架构变更 | Tech Lead + Owner | 核心维护者 |
| 发布决策 | Owner | Tech Lead + QA Lead |
| 社区规范 | Owner | 全体维护者 |
| 争议裁决 | Owner | - |

## 版本管理

### 版本号规则 (SemVer)

采用 [语义化版本 2.0.0](https://semver.org/lang/zh-CN/):

```
主版本号.次版本号.修订号
MAJOR.MINOR.PATCH
```

| 版本类型 | 触发条件 | 示例 |
|---------|---------|------|
| PATCH | Bug 修复，向后兼容 | 0.2.0 → 0.2.1 |
| MINOR | 新功能，向后兼容 | 0.2.0 → 0.3.0 |
| MAJOR | 破坏性变更 | 0.2.0 → 1.0.0 |

**特殊规则**（0.x 版本期间）:
- 0.x.y 期间，MINOR 版本变更可能包含破坏性变更
- 达到 1.0.0 后严格遵守 SemVer

### Release 周期

#### 常规发布

| 版本类型 | 周期 | 时间 |
|---------|------|------|
| PATCH | 按需 | Bug 修复后尽快 |
| MINOR | 月度 | 每月最后一个周五 |
| MAJOR | 季度/按需 | 重大更新完成时 |

#### 发布日历

```
周一          周三          周五
│            │            │
│  功能冻结   │  发布候选   │
│  (Code     │  (RC       │
│   Freeze)  │   Release) │
│            │            │
└────────────┴────────────┘
         ↓
    正式发布
   (Release)
```

#### 发布流程时间表

| 时间 | 活动 | 负责人 |
|------|------|--------|
| T-7 天 | 功能冻结，停止新功能合并 | Owner |
| T-5 天 | 开始回归测试 | QA Lead |
| T-3 天 | 发布候选版本 RC1 | Core Maintainer |
| T-2 天 | RC 测试反馈收集 | QA Team |
| T-1 天 | 修复 RC 发现的问题 | Tech Team |
| T-0 天 | 正式发布 | Owner |

### 变更日志维护

#### CHANGELOG.md 格式

```markdown
## [0.2.1] - 2024-03-15

### Fixed
- 修复配置文件路径解析错误 (#57)
- 修复内存泄漏问题 (#62)

### Changed

### Added
- 新增全文搜索功能 (#42)
- 新增配置文件热重载 (#45)

### Changed
- 优化搜索性能，提升 30%

### Fixed
- 修复 CLI 参数解析问题 (#38)
```

#### 变更日志规范

1. **每个版本必须有变更日志条目**
2. **按类型分组**: Added, Changed, Deprecated, Removed, Fixed, Security
3. **关联 Issue**: 每个变更必须关联 Issue 编号
4. **破坏性变更**: 必须单独列出 Breaking Changes 章节
5. **更新时机**: PR 合并时立即更新，而非发布时

## 发布流程

### 发布前检查清单

#### 代码质量
- [ ] 所有测试通过
- [ ] 无 clippy 警告
- [ ] 代码已格式化
- [ ] 无已知严重 Bug

#### 文档
- [ ] README 已更新
- [ ] CHANGELOG.md 已更新
- [ ] API 文档已更新
- [ ] 迁移指南已准备（如有破坏性变更）

#### 测试
- [ ] 单元测试覆盖率 > 80%
- [ ] 集成测试全部通过
- [ ] 回归测试通过
- [ ] 手动测试完成

#### 构建
- [ ] 所有平台构建成功
- [ ] 二进制文件大小正常
- [ ] 依赖项安全检查通过

### 发布步骤

1. **准备发布**
   ```bash
   # 更新版本号
   cargo bump patch  # 或 minor/major
   
   # 更新 CHANGELOG.md
   # 确保所有变更已记录
   ```

2. **创建 Release Branch**
   ```bash
   git checkout -b release/v0.2.1
   ```

3. **最终测试**
   ```bash
   cargo test
   cargo clippy -- -D warnings
   ```

4. **打标签**
   ```bash
   git tag -a v0.2.1 -m "Release v0.2.1"
   git push origin v0.2.1
   ```

5. **发布到 Cargo**
   ```bash
   cargo publish
   ```

6. **创建 GitHub Release**
   - 填写 Release 说明
   - 粘贴 CHANGELOG 内容
   - 上传二进制文件（如有）

7. **通知社区**
   - 更新项目网站
   - 发送发布通知（邮件列表、社交媒体等）

本项目采用 [Contributor Covenant 2.1](https://www.contributor-covenant.org/version/2/1/code_of_conduct/) 行为准则。

- 保持礼貌和尊重
- 提供足够的信息帮助复现问题
- 感谢他人的帮助
- 避免重复提问（先搜索现有 Issue）

#### PR 沟通
- 清晰描述变更内容
- 及时回应审核意见
- 接受建设性批评
- 感谢审核者的时间

#### 讨论区沟通
- 保持话题相关
- 避免离题讨论
- 尊重不同意见
- 不使用攻击性语言

### 争议处理

#### 争议类型

| 类型 | 处理方式 |
|------|----------|
| 技术分歧 | 技术讨论 → Tech Lead → Owner |
| 行为问题 | 警告 → 暂停 → 禁止参与 |
| 版权争议 | 法律审查 → 移除内容 |
| 安全问题 | 立即修复 → 安全公告 |

#### 处理流程

```
争议发生
    ↓
当事人尝试解决（48 小时）
    ↓
无法解决？
    │
    ├─ 是 → 提交维护者调解
    │        ↓
    │     仍无法解决？
    │        │
    │        ├─ 是 → 提交 Owner 裁决
    │        └─ 否 → 执行调解方案
    │
    └─ 否 → 争议解决
```

#### 处罚措施

| 违规程度 | 处罚 |
|---------|------|
| 轻微 | 私下警告 |
| 中等 | 公开警告，暂停参与 7 天 |
| 严重 | 暂停参与 30 天 |
| 极严重 | 永久禁止参与 |

## 决策机制

### 决策类型

| 决策类型 | 决策方式 | 参与人员 |
|---------|---------|----------|
| 日常决策 | 维护者决定 | 相关维护者 |
| 技术决策 | 讨论 + Tech Lead 批准 | Tech 团队 |
| 重大变更 | 投票（简单多数） | 核心维护者 |
| 战略决策 | Owner 决定 | Owner + 咨询团队 |

### 投票规则

#### 适用范围
- 新核心维护者任命
- 重大架构变更
- 发布策略变更
- 社区规范变更

#### 投票流程
1. 提案在 Issue 中提出
2. 讨论期 7 天
3. 投票期 7 天
4. 简单多数通过（>50%）
5. Owner 有否决权

#### 投票权
- 核心维护者：1 票
- Owner: 1 票 + 否决权
